# evergen

evergen is a tiny, generic Python codegen orchestrator. A generator is any
Python file that exposes `gen() -> str`; evergen executes it, signs the
generated body with a header, and writes the mapped output file.

Generated files are real code that you commit. The signed header makes
regeneration safe by mechanism: evergen can tell the difference between a clean
generated file, a stale generated file, and a file that was edited by hand.
When you want to hand-own a generated file, graduate it by removing the header
and deleting the generator.

Generator files are trusted code. evergen loads and executes them by file path
with the same trust model as `setup.py`; only run generators from repositories
you trust.

The determinism law: `gen()` must be deterministic. The same repository state
must produce the same bytes. evergen documents this law but does not enforce it.

## CLI

```sh
evergen --output OUT_PATTERN [--check] [--overwrite] [--header TEMPLATE] INPUT_PATTERN [INPUT_PATTERN ...]
```

`INPUT_PATTERN` is a glob containing exactly one `{}` placeholder. The text
matched by `{}` is the capture and is substituted into `OUT_PATTERN`, which
also must contain exactly one `{}`.

As a convenience, an input may also be a plain `.py` file path with no glob
characters and no `{}`. Its capture is the filename minus the final `.py` and
minus a trailing generator suffix of `.eg`, `.gen`, or `.generator` when present.
For example, `summary.eg.py` maps with capture `summary`.

Generator files are loaded by file path with `importlib` spec-from-file
machinery, never by package import. Dotted filenames such as `summary.eg.py`
work.

Output path resolution:

- If `OUT_PATTERN` is absolute, it is used as an absolute path.
- If `OUT_PATTERN` is relative and the matching `INPUT_PATTERN` contains `**`,
  the output path is resolved relative to the matched generator file's
  directory. For example, `src/**/{}.eg.py` with `--output {}__out.py` maps
  `src/pkg/user.eg.py` to `src/pkg/user__out.py`.
- Otherwise, a relative `OUT_PATTERN` is resolved relative to the current
  working directory.

Processing order is deterministic: matched generator paths are sorted before
execution. If two inputs map to the same output path, evergen exits with a hard
error before writing any file.

### Signed Header

The default first line is:

```text
# @generated by evergen from {source} — SignedHash<<{hash}>> — do not hand-edit
```

`{source}` is the generator path relative to the output file. `{hash}` is the
first 16 hex characters of SHA-256 over the generated body, excluding the
header line. Newlines are normalized to `\n` before hashing, so CRLF checkouts
do not create false dirty reports.

`--header TEMPLATE` replaces the header line. The template must be one line and
must contain `{hash}`. `{source}` is optional. Detection is
decoration-independent, but the machine-invariant marker is the literal token
`SignedHash<<...>>`; custom templates must place the digest in
`SignedHash<<{hash}>>`. evergen detects the token by scanning the first five
lines of an existing file. Hash verification is the hash of everything after
the line containing the token.

### Write Semantics

Without `--overwrite`:

- Missing target: write it.
- Target with a valid evergen header: write it.
- Target without the `SignedHash<<...>>` token: refuse with `not generated by
  evergen; use --overwrite to replace`.
- Target with the token but a hash mismatch: refuse with `hand-edited; keep
  your edits by removing the header and deleting the generator, or discard them
  with --overwrite`.

With `--overwrite`, evergen replaces unmanaged and dirty targets.

### Check Mode

`--check` writes nothing. It exits nonzero if any target is:

- `MISSING`: the output file does not exist.
- `DIRTY`: the output has an evergen token but its signed hash no longer
  matches its body.
- `STALE`: the output hash is valid, but rerunning the generator would produce
  different body bytes.

`--check` exits zero only when every target is clean.

## Examples

### 1. Trivial generator

Create a generator:

```sh
cat > hello.eg.py <<'PY'
def gen():
    return 'print("hello world")\n'
PY
```

Run evergen:

```sh
uv tool run --from /path/to/evergen evergen --output '{}.py' '{}.eg.py'
```

The output is `hello.py`:

```python
# @generated by evergen from hello.eg.py — SignedHash<<4660ab1ff310887b>> — do not hand-edit
print("hello world")
```

### 2. Jinja2 convention

evergen does not depend on Jinja2. If you want templates, make Jinja2 your
project's dependency and call it from your generator.

Layout:

```text
templates/
  model.j2.py
models.eg.py
```

`templates/model.j2.py`:

```jinja2
class {{ name }}:
    table = "{{ table }}"
```

`models.eg.py`:

```python
from pathlib import Path
from jinja2 import Template

ROWS = [
    {"name": "User", "table": "users"},
    {"name": "Invoice", "table": "invoices"},
]

def render(template_name, **data):
    text = Path("templates", template_name).read_text()
    return Template(text).render(**data)

def gen():
    return "\n\n".join(render("model.j2.py", **row) for row in ROWS) + "\n"
```

Run:

```sh
uv tool run --from /path/to/evergen evergen --output '{}.py' '{}.eg.py'
```

Pre-commit hook:

```yaml
repos:
  - repo: local
    hooks:
      - id: evergen-check
        name: evergen check
        entry: uv tool run --from /path/to/evergen evergen --check --output '{}.py' '{}.eg.py'
        language: system
        pass_filenames: false
```
